## Java Best Practices

- Import built-in Java modules first
- Import third-party libraries after built-in modules
- Import application-specific modules after third-party libraries
- Follow the naming conventions for classes, methods, and variables
- Write unit tests for all business logic
- Use `final` for constants and fields that should not be modified
- Avoid using magic numbers and hard-coded strings
- Use meaningful variable and method names
- Follow the Single Responsibility Principle (SRP) to ensure that each class has only one reason to change
- Use interfaces to define contracts for your classes
- Use dependency injection to manage dependencies between classes
- Avoid tight coupling between classes by using interfaces and dependency injection
- Use design patterns such as Factory, Singleton, and Observer where appropriate
- Use the Liskov Substitution Principle (LSP) to ensure that subclasses can be used in place of their base classes
- Use the Open/Closed Principle (OCP) to ensure that classes are open for extension but closed for modification
- Use the Interface Segregation Principle (ISP) to ensure that interfaces are small and focused
- Use the Dependency Inversion Principle (DIP) to ensure that high-level modules depend on abstractions rather than concrete implementations
- Use latest Java features and best practices to improve code readability and maintainability (Java 21)

## Spring Boot Best Practices

- Use latest Spring Boot features and best practices to improve application performance and scalability (Spring Boot 4.0.5)
- Follow Spring Boot conventions and best practices for configuration, logging, and error handling
